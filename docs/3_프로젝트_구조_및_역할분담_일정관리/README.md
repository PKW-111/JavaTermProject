## 1. 문서 목적 및 범위

본 문서는 **2주차 산출물인 요구사항 분석 및 설계 문서**를 기반으로 FreshKeeper Smart Fridge의 구조를 실제 구현 관점에서 구체화하고, 팀원별 책임 범위와 일정 관리 계획을 명확히 정리하는 것을 목적으로 한다.

초기 FreshKeeper는 콘솔 기반 프로그램으로 설계되었으나, 프로젝트 진행 과정에서 사용성과 기술적 완성도를 높이기 위해 Java Swing 기반 데스크톱 GUI 프로그램으로 확장하였다.

# 2. 프로젝트 구조 설계

## 2.1 요구사항-기능-모듈 매핑

요구사항이 실제 코드의 어떤 클래스에서 구현되는지 추적할 수 있도록 정리하였다.

| 요구사항 ID | 요구사항 요약       | 기능 단위                     | 담당 모듈 및 클래스                             | 완료 판정 기준                   |
| ------- | ------------- | ------------------------- | --------------------------------------- | -------------------------- |
| F-01    | 식재료 등록        | 입력, 검증, 객체 생성, 목록 추가      | `FreshKeeperGUI`, `FoodService`, `Food` | 정상 입력한 식재료가 테이블에 즉시 표시됨    |
| F-02    | 식재료 전체 조회     | 목록 조회 및 테이블 표시            | `FoodService`, `FreshKeeperGUI`         | 등록된 모든 식재료가 누락 없이 표시됨      |
| F-03    | 식재료 삭제        | 항목 선택, 삭제 확인, 목록 제거       | `FreshKeeperGUI`, `FoodService`         | 선택한 식재료가 확인 후 삭제됨          |
| F-04    | 남은 일수 계산      | 현재 날짜와 유통기한 차이 계산         | `Food`                                  | 현재 날짜 기준 남은 일수가 정확히 계산됨    |
| F-05    | 상태 분류         | 만료, 오늘까지, 위험, 주의, 안전 분류   | `Food`                                  | 정해진 기준에 따라 상태가 일관되게 출력됨    |
| F-06    | 메뉴 추천         | 보유 재료와 레시피 비교             | `RecipeService`, `Recipe`               | 보유 식재료를 반영한 추천 목록이 생성됨     |
| F-07    | 부족 재료 표시      | 미보유 재료 추출                 | `RecipeService`, `RecipeResult`         | 추천 결과에 부족한 재료가 표시됨         |
| F-08    | 잘못된 입력 처리     | 숫자, 날짜, 빈 입력 검증           | `FreshKeeperGUI`                        | 잘못된 입력에도 프로그램이 종료되지 않음     |
| F-09    | 데이터 저장 및 불러오기 | CSV 파일 입출력                | `FoodService`, `FreshKeeperGUI`         | 프로그램 재실행 후 기존 데이터가 복원됨     |
| F-10    | 식재료 검색        | 이름, 카테고리, 위치 검색           | `FoodService`, `FreshKeeperGUI`         | 검색어와 일치하는 항목만 테이블에 표시됨     |
| F-11    | 식재료 수정        | 선택 항목 정보 변경               | `FoodService`, `FreshKeeperGUI`         | 변경한 정보가 객체와 테이블에 반영됨       |
| F-12    | 유통기한 알림       | 만료 및 임박 식재료 확인            | `FoodService`, `FreshKeeperGUI`         | 만료·위험 식재료가 있을 때 알림창이 표시됨   |
| F-13    | 상태 요약 카드      | 상태별 식재료 개수 계산 및 표시        | `FreshKeeperGUI`, `Food`                | 만료, 위험, 주의, 안전 개수가 목록과 일치함 |
| F-14    | 추천 점수 계산      | 일치 재료, 부족 재료, 임박 재료 점수 반영 | `RecipeService`, `RecipeResult`         | 기준에 따라 추천 점수가 정확히 계산됨      |
| F-15    | 추천 메뉴 조리      | 추천 목록에서 메뉴 선택             | `FreshKeeperGUI`, `RecipeResult`        | 선택한 메뉴의 조리 기능이 실행됨         |
| F-16    | 식재료 수량 차감     | 조리에 사용된 재료 수량 감소          | `RecipeResult`, `FoodService`           | 사용된 식재료의 수량이 자동으로 감소함      |
| F-17    | 수량 0 식재료 처리   | 삭제 여부 확인 및 목록 정리          | `FreshKeeperGUI`, `FoodService`         | 수량 0인 식재료에 대해 삭제 확인창이 표시됨  |
| F-18    | Swing GUI 제공  | 입력창, 테이블, 카드, 버튼 화면 구성    | `FreshKeeperGUI`, `RoundedPanel`        | 모든 주요 기능을 GUI에서 실행할 수 있음   |

## 2.2 기능 분해 구조

| 상위 기능   | 하위 기능                  | 입력과 출력               | 선행 조건        | 비고            |
| ------- | ---------------------- | -------------------- | ------------ | ------------- |
| 식재료 관리  | 등록, 조회, 검색, 수정, 삭제     | 사용자 입력 ↔ 식재료 목록      | 프로그램 실행      | 핵심 사용자 기능     |
| 유통기한 관리 | 남은 일수 계산, 상태 분류, 알림    | 유통기한 → 남은 일수 및 상태    | 유효한 날짜 입력    | 조회와 추천의 공통 기준 |
| 상태 대시보드 | 상태별 개수 계산 및 표시         | 식재료 목록 → 상태 카드       | 식재료 데이터 존재   | 등록·수정·삭제 후 갱신 |
| 메뉴 추천   | 재료 비교, 부족 재료 계산, 점수화   | 식재료 목록 + 레시피 → 추천 결과 | 식재료 1개 이상 등록 | 점수 높은 순서로 표시  |
| 메뉴 조리   | 메뉴 선택, 재료 수량 차감        | 추천 메뉴 → 변경된 재고       | 추천 메뉴 존재     | 수량 0 처리 포함    |
| 데이터 지속성 | CSV 저장 및 불러오기          | 메모리 ↔ CSV 파일         | 저장 형식 일치     | 종료 후에도 데이터 유지 |
| GUI 화면  | 입력창, 테이블, 상태 카드, 추천 카드 | 사용자 이벤트 → 기능 실행      | Swing 실행 환경  | 스마트 냉장고 패널 형태 |
| 안정성     | 입력 검증, 예외 처리, 안내창      | 잘못된 입력 → 오류 안내       | 없음           | 비정상 종료 방지     |

## 2.4 시스템 구조도

```text
사용자
  ↓
Main.java
  ↓
FreshKeeperGUI.java
  ├── FoodService.java
  │     ├── Food.java
  │     └── data/foods.csv
  │
  ├── RecipeService.java
  │     ├── Recipe.java
  │     └── RecipeResult.java
  │
  └── RoundedPanel.java
```

### 구조 설명

* `Main.java`는 프로그램의 시작점이며 GUI 실행을 담당한다.
* `FreshKeeperGUI.java`는 화면 구성과 사용자 이벤트 처리를 담당한다.
* `FoodService.java`는 식재료 목록과 CSV 파일을 관리한다.
* `Food.java`는 식재료 데이터와 유통기한 계산을 담당한다.
* `RecipeService.java`는 메뉴 추천 점수를 계산한다.
* `Recipe.java`는 메뉴명과 필요한 재료를 저장한다.
* `RecipeResult.java`는 추천 결과와 조리 기능을 처리한다.
* `RoundedPanel.java`는 상태 카드와 메뉴 추천 카드의 화면 표현을 돕는다.

> `RoundedPanel`이 별도 파일이 아니라 `FreshKeeperGUI.java` 내부 클래스이면 실제 코드 구조에 맞게 표기한다.

## 2.5 실행 시퀀스

### 시나리오 A: 프로그램 시작

```text
프로그램 실행
→ Main에서 FreshKeeperGUI 생성
→ FoodService가 foods.csv 읽기
→ 식재료 목록 복원
→ JTable 갱신
→ 상태 카드 개수 갱신
→ 유통기한 알림 표시
```

### 시나리오 B: 식재료 등록

```text
사용자 입력
→ FreshKeeperGUI에서 입력값 검증
→ Food 객체 생성
→ FoodService에 추가
→ JTable 갱신
→ 상태 카드 갱신
→ 등록 성공 메시지 출력
```

### 시나리오 C: 식재료 수정

```text
테이블에서 식재료 선택
→ 수정할 값 입력
→ 입력값 검증
→ FoodService에서 객체 정보 변경
→ JTable 및 상태 카드 갱신
→ 수정 성공 메시지 출력
```

### 시나리오 D: 메뉴 추천

```text
보유 식재료 목록 확인
→ RecipeService에서 레시피 비교
→ 보유 재료와 부족 재료 계산
→ 유통기한을 반영한 점수 계산
→ RecipeResult 생성
→ 추천 카드 화면 출력
```

### 시나리오 E: 메뉴 조리

```text
추천 메뉴 선택
→ RecipeResult에서 필요 재료 확인
→ FoodService의 식재료 수량 차감
→ 수량이 0인 식재료 확인
→ 삭제 여부 질문
→ 테이블, 상태 카드, 추천 결과 갱신
```

### 시나리오 F: 저장 및 재실행

```text
저장 버튼 또는 프로그램 종료
→ FoodService가 목록을 CSV 형식으로 저장
→ 프로그램 종료
→ 다음 실행 시 foods.csv 불러오기
→ 이전 식재료 목록 복원
```

## 3.1 팀원별 업무 및 책임 범위

## 3.4 충돌 및 일정 지연 대응 규칙

1. 기능 충돌 발생 시 핵심 요구사항의 정상 동작을 우선한다.
2. 동시에 같은 파일을 수정하지 않도록 작업 전 담당 파일을 공유한다.
3. 병합 충돌이 발생하면 마지막 커밋을 덮어쓰지 않고 두 작업 내용을 비교한다.
4. 일정이 지연되면 새로운 기능 추가보다 기존 기능 안정화를 우선한다.
5. GUI 디자인보다 등록, 조회, 저장, 추천 등 핵심 동작을 우선한다.
6. 설계 변경 시 요구사항, 구조 문서, 기능 구현 문서를 함께 수정한다.
7. 해결되지 않은 오류는 숨기지 않고 GitHub Issue에 기록한다.

# 6. 결론

FreshKeeper Smart Fridge는 초기 콘솔 기반 식재료 관리 프로그램에서 Java Swing 기반 데스크톱 GUI 프로그램으로 확장되었다.

프로젝트는 식재료 등록·조회·검색·수정·삭제, 유통기한 계산과 상태 분류, CSV 저장 및 불러오기, 메뉴 추천, 부족 재료 표시, 조리 후 수량 차감 기능을 하나의 흐름으로 연결하였다.

요구사항과 코드 모듈을 추적할 수 있도록 구조를 설계하였으며, 두 명의 팀원이 핵심 기능 구현과 안정화·테스트·문서화를 나누어 담당하도록 역할을 구분하였다.

또한 단계별 일정, 마일스톤, WBS, 리스크 대응, 품질 게이트를 통해 최종 구현과 발표까지 체계적으로 관리한다.
